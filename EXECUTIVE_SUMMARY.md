## Subagent perspectives

### Python Expert
**Praise**: Well-structured changes. `install_guides_dir()` follows the existing `screenshot_dir()` pattern. Filename sanitization with `re.sub(r'[^a-zA-Z0-9._-]', '_', package_name)` prevents path traversal. The `from src.config import install_guides_dir` local import inside `_install_package` avoids circular imports — reasonable given the tool module pattern.

**Flag**: The `_install_package` function uses a local import (`from src.config import install_guides_dir`). While pragmatic, consider moving it to module level as a lazy import pattern if more functions need it. The `subprocess` import is still needed for `_search_packages` — verified.

**Suggestion**: Consider adding type annotations to the new functions for consistency with the rest of the codebase. Also, error handling for write failures (disk full, permissions) would make the function more robust — currently `path.write_text(content)` can raise `OSError` uncaught.

### Security Expert
**Praise**: This change completely eliminates two high-risk code paths: the `pkexec pacman -S --noconfirm` privilege escalation (C-1) and the `YAY_ANSWER_ALL` automatic confirmation (H-3). No new subprocess calls were introduced. The filename sanitization regex is solid — only alphanumeric, dots, underscores, and hyphens survive, which prevents path traversal injection. The guidance text in the MD file explicitly tells users to verify packages before installing.

**Flag**: The MD file content includes the `package_name` directly (unsanitized) in the `sudo pacman -S {package_name}` and `yay -S {package_name}` commands. If the LLM hallucinates a package name containing shell metacharacters (e.g., `pkg; rm -rf /`), this would be a command injection risk when the user runs it. The sanitization is for the filename only.

**Suggestion**: Consider adding a warning about the LLM potentially being wrong about package names, and advise users to search themselves. The existing `*Generated by GnomePilot — always verify packages before installing.*` footer partially addresses this.

### AI Integration Expert
**Praise**: The test updates are minimal and focused. All existing assertions that depend on the old string values were updated. The `test_skill_manifest` tests for `test_all_enabled_builds_tool_list` and `test_disabled_skill_omitted` pass with the new strings.

**Flag**: All 7/7 test suites pass when run from the venv (`source .venv/bin/activate && python3 run_tests.py --unit`).

**Suggestion**: No new tests were added for `install_guides_dir()` or the rewritten `_install_package`. The plan's Phase 2 mentions a new `test_package_manager.py` with 3 tests — these were not created. Consider adding a `test_config.py` for the new config function.

### OS/Linux Expert
**Praise**: The `install_guides/` directory is created with `parents=True, exist_ok=True` — good default behavior. Using `Path` from pathlib is idiomatic and portable. The `.gitignore` addition is correct.

**Flag**: The config uses a relative path `"install_guides"` — this resolves to `PROJECT_DIR / "install_guides"` which is correct for development but note that running from a different working directory won't affect it since we use `PROJECT_DIR`.

**Suggestion**: Ensure the `install_guides/` directory has appropriate permissions. Since `mkdir(parents=True)` uses default umask, it should be fine for a single-user setting.
